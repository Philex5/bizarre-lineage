# Adsterra 广告形式与收益调研

日期：2026-03-13

## 结论摘要

对于 `bizarre-lineage` 这种以 SEO 内容、攻略页、百科页、资讯页为主的内容站，Adsterra 的接入结论可以直接归纳为三句话：

1. 如果只看绝对收益，`Popunder` 通常是 Adsterra 官方反复强调的高收益格式。
2. 如果要求“侵入式较小，同时尽量把收益做高”，最优方案不是单挂 `Banner`，而是 `Social Bar + 少量 Native Banner`。
3. 不建议首发接入 `Popunder` 或 `Interstitial`，因为它们虽然更赚钱，但会明显增加打扰感，并且更容易伤害内容站的留存、品牌感知和 SEO 流量体验。

## Adsterra 主要广告形式

### 1. Popunder

表现形式：
用户点击页面任意区域后，在新标签页或新窗口打开广告页。

特点：

- 官方明确把它描述为高收益格式之一。
- 对页面布局“零占位”。
- 更偏桌面端，也支持部分移动端。
- 对用户打扰感明显高于内容内广告。

适合：

- 下载站、工具站、流量套利站、影视/娱乐站
- 愿意用体验换收益的站点

不太适合：

- 需要长期品牌信任的攻略站、文档站、教程站
- 首屏体验和留存很重要的 SEO 内容站

### 2. Social Bar

表现形式：
类似通知、聊天气泡、浮层小组件、in-page push 的轻量浮动广告。

特点：

- 官方将其定位为“UX-friendly”“non-intrusive”。
- 不需要用户订阅，可覆盖几乎全部流量。
- 兼容桌面和移动端。
- 官方多次强调 CTR 明显高于传统 web push，且不需要预留版位。

适合：

- 内容站
- 移动流量占比较高的站点
- 想尽量减少页面割裂感的站点

### 3. In-Page Push

表现形式：
页面内通知式广告，外观类似浏览器通知。

特点：

- 官方常把它作为 Social Bar 的子类或近似能力来讲。
- 覆盖面广，不依赖浏览器订阅。
- 收益和体验通常介于 Banner 与更强势格式之间。

适合：

- 需要轻量化变现、又不想过度侵入的流量页

### 4. Interstitial

表现形式：
全屏插页广告。

特点：

- 视觉冲击强，收益能力通常不错。
- 对阅读流打断最明显。

适合：

- 跳转页、分页切换、工具结果页

不太适合：

- 长文攻略页、百科页、教程页

### 5. Native Banner

表现形式：
嵌入内容流、段落间、侧边栏、相关推荐区域的原生/半原生广告。

特点：

- 侵入性最低之一。
- 收益通常低于 Popunder 和 Social Bar。
- 最适合作为补充收入，不适合独立承担主要收入目标。

适合：

- 内容站
- SEO 着陆页
- 长文页和资讯页

### 6. Display Banner

表现形式：
传统横幅、矩形展示广告。

特点：

- 最常规。
- 体验可控。
- 但普遍存在 banner blindness，通常不是 Adsterra 体系里收益最强的格式。

### 7. Smartlink / Direct Link

表现形式：
把流量导向一个智能跳转链接。

特点：

- 更适合站外流量、社媒流量、工具跳转流量。
- 不适合作为内容站主站内广告方案。

## 收益判断

### 绝对收益能力排序

结合 Adsterra 官方描述、官方案例和第三方评测，可以把“常见收益能力”粗略理解为：

`Popunder > Social Bar > Interstitial ~= In-Page Push > Native Banner > Display Banner`

说明：

- 这是内容站场景下的经验排序，不是固定 eCPM 承诺。
- 官方也多次强调收益会受 GEO、设备、垂类、广告主竞争、用户行为、流量质量影响。

### 官方能确认到的收益信号

1. 官方首页与 Publishers 页面都把 `Popunder` 直接归类为高收益格式。
2. 官方将 `Social Bar` 定义为更友好但仍具高 CPM 潜力的格式，并强调其 CTR 可显著高于传统 web push。
3. 官方案例里，内容型/资讯型站点常见的组合不是“纯 Banner”，而是 `Banner + Social Bar` 或 `Popunder + Social Bar + Native`。
4. 官方 Google Discover 新闻站案例中，最终稳定方案是 `Popunder + Social Bar + Native`，并给出巴基斯坦流量 CPM 约 `$1.8-$2.0`、周收入 `1000+ 美元` 的案例，但该案例依赖 `150,000+` 日曝光量。

### 官方列出的部分 CPM 区间

官方 `Best CPM Rates by Country` 页面给出了截至 2026-03-09 的估算值，波动很大，但可以拿来判断量级。

以“主流网站流量 + Popunder/Direct Link”为例，官方列出的部分站点流量参考值：

- 美国：Android 约 `$3.2 CPM`，iOS 约 `$4.6 CPM`，Windows 约 `$4.5 CPM`
- 加拿大：Android 约 `$2.3 CPM`
- 英国：Android 约 `$1.9 CPM`

这些数字只能作为“高质量 GEO 有明显溢价”的参考，不能直接当成你的站点收益预测。

## 低侵入 + 高收益的最佳接入方案

### 推荐方案

`主方案：Social Bar + 少量 Native Banner`

这是当前最符合你项目方向的方案。

原因：

1. `Social Bar` 是 Adsterra 自己强调的“更友好、覆盖全流量、收益强于普通 banner”的格式。
2. 它不需要像 Banner 一样抢页面黄金位置，也不会像 Interstitial 或 Popunder 一样强打断阅读。
3. `Native Banner` 可以放在内容流的自然断点，补充广告曝光而不明显破坏阅读连续性。
4. 对于游戏攻略站、codes 页、tier list 页、stand 详情页，这种组合比纯展示广告更赚钱，也比 Popunder 更稳妥。

### 为什么不是 Popunder

虽然 `Popunder` 很可能是绝对收益更高的单一格式，但它不满足“侵入式较小”这个约束。

对你这个项目，它的主要问题是：

- 用户首次点击就可能触发新标签页，打断明显
- 对品牌观感伤害更大
- 对长线 SEO 内容站不够友好
- 容易拉低回访用户体验

所以结论不是“Popunder 不赚钱”，而是“它不适合作为这个站的默认首发方案”。

### 为什么不是只上 Native Banner / Display Banner

因为这样虽然最温和，但很可能放弃了 Adsterra 最大的变现优势。

如果只上 Banner：

- 收益天花板更低
- 更依赖高可视版位
- 更容易被用户无视

所以更优解不是“最轻”，而是“轻量主力 + 低风险补充”。

## 推荐落地方式

### 第一阶段

全站接入：

- `Social Bar` 作为主收益格式

仅在高价值内容模板接入：

- `Native Banner`

建议优先页面：

- `codes` 页
- `tier-list` 页
- `guides/*` 页
- `stands/*` 页
- `posts/*` 资讯页

### Native Banner 建议位置

每页最多 1 到 2 个，先少量测试：

- 标题和首段之间不要放
- 第一屏正文中段可放 1 个
- 长文 40% 到 60% 阅读深度之间可放 1 个
- 侧边栏如果移动端权重高，则优先不依赖侧边栏收益

### 不建议首发的位置

- 首屏 hero 正下方紧贴广告
- 目录上方强插广告
- 每个 section 后都插广告
- 连续两个广告位

## 风险与取舍

### 用户体验

最优平衡点：

- `Social Bar` 开启
- `Native Banner` 控制密度
- 不开 `Popunder`
- 不开 `Interstitial`

### SEO 与页面质量

Adsterra 官方提到 Social Bar 代码较轻，对页面下载速度影响较小，但这不等于你可以忽略性能验证。

建议关注：

- CLS 是否上升
- LCP 是否变差
- 移动端首屏交互是否被遮挡
- Google Discover / Search Landing Page 退出率是否明显恶化

### 商业风险

Adsterra 的高收益通常来自广告主更激进的投放需求，因此你需要特别关注：

- 广告素材观感是否影响站点品牌
- 某些 GEO 下的创意是否过于夸张
- 是否需要找账户经理限制不想出现的广告类型

## 推荐的 A/B 测试顺序

### 方案 A

- 仅开 `Social Bar`

目标：

- 测试对 RPM、停留时长、跳出率的影响

### 方案 B

- `Social Bar + 1 个 Native Banner`

目标：

- 测试是否在用户体验变化有限的前提下提升页面 RPM

### 方案 C

- `Social Bar + 2 个 Native Banner`

目标：

- 找到收益与体验的平衡上限

### 不建议默认进入的方案

- `Popunder + Social Bar`
- `Interstitial + Social Bar`

这两个方案更适合在你明确愿意用体验换收入时，再单独灰度到某些流量页或低品牌敏感页。

## 收款与结算

根据官方资料，当前可确认的结算信息包括：

- 最低提现可低至 `$5`，取决于支付方式，`Paxum` 和 `WebMoney` 可达 `$5`
- Local Bank Transfer 典型门槛为 `$25`
- 传统 Wire Transfer 门槛明显更高，可到 `$1000`
- 自动结算节奏为 `NET15`，通常每月两次

参考来源：

- [官方：Minimum Payout](https://adsterra.com/blog/adsterra-minimum-payout-for-publishers/)
- [官方：Payouts in Local Currency](https://adsterra.com/blog/payouts-in-local-currency/)

## 最终建议

如果你的目标是：

`侵入式较小 + 收益尽量大 + 不破坏内容站定位`

那就采用下面这个方案：

### 最终推荐

- 主收益格式：`Social Bar`
- 补充收益格式：`Native Banner`
- 首发禁用：`Popunder`
- 首发禁用：`Interstitial`
- 站外流量单独考虑：`Smartlink`

一句话总结：

`Popunder` 是更像“暴力变现”的高收益方案，`Social Bar + Native Banner` 才是更适合 Bizarre Lineage 这类游戏攻略内容站的“低侵入高收益方案”。

## 后续可执行项

如果需要进入实施阶段，建议下一步做这三件事：

1. 在站点模板中预留广告插槽开关，只对 `codes`、`tier-list`、`guides`、`posts` 模板开放。
2. 用环境变量控制 Adsterra script 是否注入，先在预览环境灰度。
3. 记录广告实验指标：页面 RPM、广告可见率、跳出率、平均互动时长、Discover/SEO 页面自然流量波动。

## 最终页面放置规范

以下规范基于当前已确认的投放策略：

- 桌面端默认使用 `Native Banner 1x4`
- 移动端默认降低密度处理
- 首页不放首屏 Hero 下方，改放中段模块之后
- `stands` 需要区分列表页和详情页

### 首页 ` / `

推荐结构：

```text
[Header / Nav]
[Hero]
[模块 1: 热门攻略 / 最新内容]
[模块 2: 新手入口 / 热门页面]
[Ad Slot A: 1x4 Native]
[模块 3: Tier List / Codes / 热门 Stand]
[模块 4: 最新文章]
[Ad Slot B: 可选]
[Footer]
```

建议：

- 桌面端优先放 `1 组`
- 如果首页模块足够长，可加第 `2` 组
- 第一组放在首页中间一点的位置，也就是第 `2` 个主要模块之后
- 第二组如果启用，放在更靠后的位置，不要连续贴模块放
- 移动端默认只放 `1 组`

### 兑换码页 ` /codes `

推荐结构：

```text
[H1 + 简介]
[兑换码状态说明 / 更新时间]
[Codes 主表格或主列表]
[Ad Slot A]
[兑换码使用教程 / FAQ]
[Ad Slot B: 可选]
[相关文章 / Footer]
```

建议：

- 不要把广告插进 codes 主表格中间
- 桌面端可放 `1-2 组`
- 移动端建议 `1 组`

### Tier List 页 ` /tier-list `

推荐结构：

```text
[H1 + 简介]
[Tier List 总览 / 评分说明]
[完整 Tier 表]
[Ad Slot A]
[各 Tier 详细说明]
[Ad Slot B: 可选]
[FAQ / 推荐阅读]
```

建议：

- 广告放在总表之后，不要拆开 `S/A/B/C` 分层
- 桌面端可放 `1-2 组`
- 移动端建议 `1 组`

### 攻略页 ` /guides/* `

推荐结构：

```text
[H1 + 摘要]
[目录 / 导航]
[正文 Section 1]
[正文 Section 2]
[Ad Slot A]
[正文 Section 3]
[正文 Section 4]
[Ad Slot B: 长文可加]
[FAQ / 相关阅读]
```

建议：

- 第一组放在正文约 `25%-35%`
- 第二组放在正文约 `60%-70%`
- 短文只放 `1 组`
- 不要贴着目录和首段放

### Stands 列表页 ` /stands `

推荐结构：

```text
[H1 + 简介]
[筛选 / 分类导航]
[Stand 列表第一段]
[Ad Slot A]
[Stand 列表第二段]
[Ad Slot B: 可选]
[FAQ / Footer]
```

建议：

- 放在一段 stand 卡片列表之后
- 不要放在筛选器正下面
- 桌面端可放 `1-2 组`
- 移动端建议 `1 组`

### Stand 详情页 ` /stands/* `

推荐结构：

```text
[H1 + 简介]
[基础信息卡: Tier / Best For / How To Get]
[Ad Slot A]
[技能 / 强度 / 优缺点解析]
[获取方法 / 搭配建议]
[Ad Slot B: 长文可选]
[FAQ / 相关阅读]
```

建议：

- 第一组放在“基础信息卡”和“长文解析”之间
- 第二组仅在正文明显较长时启用
- 不要贴着 `H1`
- 不要插进属性表格或核心信息块中间

### 文章页 ` /posts/* `

推荐结构：

```text
[H1 + 封面 + 摘要]
[正文前几段]
[Ad Slot A]
[正文中后段]
[Ad Slot B: 长文可加]
[结尾 / 相关阅读]
```

建议：

- 不要首段后立刻放
- 最好在第 `3-5` 段后放第一组
- 长文再加第二组

### 术语页 ` /terms/* `

推荐结构：

```text
[H1 + 定义]
[核心解释]
[Ad Slot A]
[获取方式 / 用法 / 关联词条]
[Footer]
```

建议：

- 这类页通常偏短，默认只放 `1 组`
- 不建议放第 `2` 组

## 数量上限建议

桌面端：

- 首页：`1-2 组`
- `codes` / `tier-list` / 长攻略 / 长文章：`1-2 组`
- `stands` 列表页：`1-2 组`
- `stands` 详情页 / `terms` / 短文：`1 组`

移动端：

- 首页：`1 组`
- 其他页面默认：`1 组`
- 如未获得更轻量的移动端样式，不建议轻易上第 `2` 组
